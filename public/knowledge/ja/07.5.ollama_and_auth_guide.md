# ローカル LLM とクラウド認証の統合戦略：Ollama 導入と Cookie 抽出のハック

## @Overview

AI エンジンの開発において、計算リソースの選択は重要です。コストとプライバシーに優れた「ローカル推論」と、最高知能を誇る「クラウド API」のどちらを選ぶべきか。Moyind プロジェクトでは、これらを動的に切り替えるハイブリッド構成を採用しています。本稿では、Ollama によるローカル環境構築と、クラウドサービスの認証情報を安全に抽出する手法について解説します。

---

## 1. ローカル LLM 中枢：Ollama 推論サーバーの構築

NVIDIA RTX シリーズ等の GPU を所有している場合、プライバシー保護とコスト削減のために Ollama によるローカル推論環境の構築を推奨します。

### 📦 VRAM 容量に応じたモデル選定基準

搭載されているビデオメモリ（VRAM）の容量が、動作可能なモデルの知能に直結します：

- **2-4GB**: `llama3.2:3b`。軽量ですが、API フォーマットのテストや単純な対話には十分です。
- **6-8GB**: **推奨環境**。`qwen2.5:7b` が最適です。日本語の語彙力と論理構成に優れています。
- **12GB+**: `qwen2.5:14b` など、より推論精度の高いモデルが快適に動作します。
- **24GB (RTX 3090/4090)**: `32b` クラスや、最新のプレビュー版モデルによる高度な論理推論が可能です。

---

## 2. クラウド認証の活用：Firefox Cookie 抽出スキーム

Gemini 1.5 Pro などの最強モデルを開発フェーズで利用したい場合、公式 API の制限を回避するためにウェブ版のセッションを活用することがあります。

### 🦊 なぜ Firefox なのか？

Chrome は Cookie を OS 固有のハードウェアキーで暗号化しており、外部スクリプトからの抽出が困難です。一方、Firefox は SQLite 形式でデータを管理しているため、適切なプロトコルを用いれば認証 Token を安全に抽出できます。

1.  ブラウザで対象サービスにログイン。
2.  スクリプトが Firefox のプロファイルディレクトリから `session_id` 等を抽出。
3.  抽出されたトークンを API Gateway に渡し、プログラムからの自動アクセスを実現します。

---

## 3. レスポンスの極大化：ネイティブ Context 伝送技術

ローカルの Ollama を利用する際、一般的な OpenAI 互換レイヤーを介すとオーバーヘッドが発生します。Moyind では `/api/chat` エンドポイントを直接叩くことで、ミリ秒単位の応答速度を追求しています。

- **KV Cache の再利用**: 会話のたびに過去ログを再送信するのではなく、Ollama から返される `context` ID を保持し、次回のリクエストに添付します。
- **効果**: 推論開始までの待ち時間が 3 秒から 0.5 秒以下へと劇的に短縮され、リアルタイムな対話体験が可能になります。

---

## 💡 実務への応用アドバイス

ハイブリッドなモデル接続層を設計する際は、以下の機能を実装してください：

- **自動フォールバック**: VRAM 残量が一定値を下回った場合、自動的に軽量モデル（llama3.2:3b 等）へ切り替えるロジックを組み込むことで、システムのクラッシュを未然に防ぎます。
- **セッション管理の自動化**: セッション Cookie の有効期限を常に監視し、期限が切れる前に自動で抽出・更新するボットを立てることで、運用の手間を最小限に抑えることができます。
