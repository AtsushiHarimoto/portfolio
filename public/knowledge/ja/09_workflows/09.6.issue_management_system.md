# 任務管理系統設計與實施

> **類型**: 工作流程 / 專案管理  
> **日期**: 2026-01-11  
> **來源**: 對話 - 任務管理系統建立

---

## 摘要

建立一套類似 JIRA 的輕量化任務管理系統，統一收編所有開發任務、系統 Bug 與功能需求，解決過往 Scope 分散與入口混亂之痛點。

---

## 背景

### 歷史痛點

1. **Scope 分散管理**: 各子系統過往獨自在其 `docs/scope/` 內進行管理，缺乏全域統一視角。
2. **任務入口混淆**:
   - `workspace/ideas` 內堆積過多雜項。
   - 來自通訊對話的臨時性任務請求。
   - 缺乏標準化的任務接收與分發站。
3. **缺乏追蹤機制**:
   - 欠缺無縫銜接的「待辦清單 (TODO)」視圖。
   - 難以精確掌握哪些任務處於執行中、已結案或已封存狀態。
   - 歷史任務難以快速回溯與調閱。

### 設計目標

- 以單一入口 (Single Point of Entry) 統籌所有任務。
- 實現極簡化的狀態流轉 (State Transition) 流程。
- 確保最小化操作成本，同時易於全局檢索。
- 無縫整合現行的 Scope 管理系統。

## 詳細內容

### 目錄結構

```
workspace/
├── issues/                      # 統一任務管理根目錄
│   ├── README.md               # 系統使用指南與規範
│   ├── INBOX/                  # 新任務收件匣 (待分配)
│   ├── TODO/                   # 待辦清單 (已規劃，尚未動工)
│   ├── DOING/                  # 執行中
│   ├── DONE/                   # 已完成 (等待歸檔)
│   └── ARCHIVE/                # 歷史歸檔區 (按月份分類)
│       └── {YYYY-MM}/
├── dashboard/                   # 任務統計儀表板
│   └── README.md
└── progress/                    # 大型任務進度追蹤區
    └── {TASK_ID}_1.progress.md
```

### 統一命名規範

```
ISSUE-{SEQ}-{SCOPE}-{TYPE}-{SHORT_DESC}

參數說明:
- SEQ: 系統流水號 (001, 002...)
- SCOPE: API | WEB | GAME | TEST | META
- TYPE: BUG | FEAT | DOC | REFACTOR | TASK
- SHORT_DESC: 2-3 個單詞之精簡描述 (全英文，採 snake_case)

標準範例:
- ISSUE-042-GAME-BUG-save_load_fail
- ISSUE-043-WEB-FEAT-editor_ui
- ISSUE-044-META-DOC-workflow_update
```

### 狀態流轉矩陣

```
INBOX → TODO → DOING → DONE → ARCHIVE/{時間戳月份}/
```

### 任務目錄標準組件

```
ISSUE-XXX-XXX-XXX-XXX/
├── README.md          # 【強制必備】任務概要說明
├── 1.PLAN.md            # 【選擇性】深度架構規劃
├── 2.PROGRESS.md        # 【強烈建議】執行進度日誌
└── VERIFY.md          # 【強烈建議】驗收與測試紀錄
```

### 配套工作流 (Workflows)

1. **`/new_issue`**: 建立新任務單
   - 透過互動式問答採集需求細節。
   - 自動配發下一組可用編號。
   - 自動生成標準目錄與架構化 README。

2. **`/move_issue`**: 推移任務狀態
   - 支援 ID 模糊匹配檢索。
   - 自動更新目標 README 與狀態流轉歷史。
   - 內建狀態攔截器與防呆提醒。

3. **`/list_issues`**: 檢索任務矩陣
   - 支援多維度條件過濾 (Status / Scope / Type / Priority)。
   - 提供兩種資料呈現視野 (摘要版 / 詳盡版)。
   - 具備 Dashboard 同步刷新機制。

4. **`/archive`**: 任務與代碼深度歸檔
   - 支援三種執行模式 (Issues Only / Code Only / Full Archive)。
   - 同步處置 issues 與 scope 依賴。
   - 執行關聯性查核，並提供一併歸檔之選項。

### Dashboard 看板與儀表功能

- 總體狀態分佈 (各狀態之任務承載量子)。
- 依 SCOPE 呈現之統計分佈。
- 依 TYPE 呈現之統計分佈。
- 近期活動軌跡 (Recent Activity Log)。
- 快捷超連結矩陣。

```
/new_issue
→ 觸發互動問答以收集任務特徵
→ 系統分配 ISSUE-{下一位編號}

```


### 檢索查詢 (Query)

```
/list_issues --status DOING --scope GAME
→ 條列出當下所有隸屬 GAME 體系之進行中任務

```

```
/archive
→ 指定操作模式 (Issues / Code / Full)

→ 連動處理所屬之 Scope
```

4. **追本溯源**: 具備不可竄改之流轉歷史及關聯文檔鏈。
5. **高度自動化**: 工作流 (Workflows) 徹底覆蓋任務全週期。

## 操作風險與防護注意

- ⚠️ **命名嚴格規範**: `SHORT_DESC` 欄位強制限用英文並以 snake_case 呈現。
- ⚠️ **狀態遞進紀律**: 強烈不建議跳躍式流轉 (如自 INBOX 暴力直推至 DONE)。
- ⚠️ **定期歸檔洗脫**: 建議於每月期末執行 `/archive` 以維持系統輕量。
- ⚠️ **README 資訊健全度**: 務必確保個別 ISSUE 之 README 填載完整、無資訊斷層。
- ⚠️ **Scope 關聯檢查**: 歸檔時必須同步查核是否有對應牽連之 Scope 目錄需連帶封裝。
