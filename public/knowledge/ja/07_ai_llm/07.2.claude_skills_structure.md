# Claude Code スキル拡張の設計標準：ディレクトリ構成と軍事級の規律

## @Overview

Claude Code や Antigravity のような AI エージェントに「独自のスキル（カスタム・インストラクション）」を追加する際、単なるテキストファイルの羅列では不十分です。本稿では、Moyind プロジェクトで採用されている、エージェントのコンテキスト検知能力を最大限に引き出すためのフォルダ構成と設計基準について解説します。

---

## 1. 初心者が陥る「単一ファイル」の罠

開発効率を重視するあまり、すべての指示を一つの Markdown ファイルに詰め込んでしまうのは、現代の AI エージェントの走査メカニズムにおいて致命的な誤りです。

- **鉄則**: スキル（Skill）ごとに独立したディレクトリを持たせること。
- **利点**: ディレクトリ単位でカプセル化（Encapsulation）することで、エージェントは特定のスキル（例: `@new_issue`）が呼び出された際、そのディレクトリ内の背景情報やテンプレートを一括して読み込むことができます。

---

## 2. 標準的なスキル構成（Skills Layout）

エージェントが瞬時に「スキルの役割」を認識し、メモリキャッシュにロードするための階層構造は以下の通りです。

```text
.agent/skills/          # スキルのベースディレクトリ
├── handoff/            # 各スキルごとにフォルダを分ける
│   ├── SKILL.md        # 【重要】スキルのメイン定義書
│   └── TEMPLATE.md     # 補助的なテンプレートファイル
│
├── new_issue/          # 「@new_issue」に対応
│   └── SKILL.md
│
└── dev/                # 「@dev」に対応
    └── SKILL.md
```

### なぜディレクトリを分けるのか？

高度なスキル（例：`handoff`）は、単一の指示だけでなく、出力形式（JSON スキーマ等）やリファレンス用テンプレートを必要とします。ディレクトリとしてパッケージ化することで、エージェントは必要なコンテキストを漏れなく、かつ効率的に取得可能になります。

---

## 3. `SKILL.md`：スキルの「アイデンティティ」

各スキルディレクトリ内には、必ず `SKILL.md` という名称のファイルを配置します。このファイルの先頭に記述する **YAML Frontmatter** が、エージェントに対する「呼び出し暗号」となります。

```markdown
---
name: new_issue
description: "新しい ISSUE の作成を支援します。トリガー：@new_issue"
---
```

### 必須チェックポイント

1.  **`name`**: ディレクトリ名と完全に一致させる必要があります。これが呼び出しコマンドの識別子となります。
2.  **`description`**: 一行でその機能を表し、最後に必ず「トリガー：@xxx」と明記してください。これはエージェントの走査エンジンに対する重要な意味的ヒントとなります。

---

## 💡 実務への応用アドバイス

エンジニアとして新しい自動化スキルをエージェントに実装する際は、以下のステップを遵守してください：

- **SOP (標準作業手順書) の実装**: `SKILL.md` 内には、エージェントが実行すべき具体的な手順を SOP 形式で記述します。曖昧な指示を避け、決定論的な動作（Deterministic behavior）を促す記述を心がけてください。
- **同期スクリプトの活用**: メインリポジトリ内の `.agent/skills/` を、適宜ビルド環境や AI ツールパスへ同期させるインターフェースを用意することで、開発環境全体の統合性を保つことができます。
