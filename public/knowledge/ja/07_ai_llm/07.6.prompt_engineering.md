# AI を支配する権能：プロンプトエンジニアリングの極意 (Prompt Engineering)

## @Overview

大規模言語モデル (LLM) を扱う「プロンプトエンジニアリング」は、単なる文学的な表現の追求ではありません。真の AI ソフトウェア開発において、プロンプトは「エンジニアリング制御」そのものです。LLM という、高い演技力を持ちながらも常識を欠く「即興劇の役者」に対し、厳格な台本（制約）を課すための設計理論について解説します。

---

## 1. プロンプトの二軌道制：System と User の役割分担

API 通信において、指示は以下の 2 つの次元に明確に切り分けられます。これらを混同してはいけません。

### 👑 System Prompt (システムプロンプト)

これは「監督の隠しマイク」です。ユーザーには見えないバックエンド側に配置されます。

- **権威性**: 最も高い優先順位を持ち、AI の振る舞いの根本的なルール（例：「高慢な性格を維持せよ」「回答は JSON 形式のみで行え」等）を定義します。
- **防御線**: ユーザーがいかなる誘導を行っても、AI が守るべき「底線（ボトムライン）」を確立します。

### 🗣️ User Prompt (ユーザープロンプト)

これは「舞台上での物語の進展」です。

- **動的コンテキスト**: ユーザーの発言や、システムが注入する環境パラメータ（例：【現在の環境】夜 10 時、嵐の夜）を伝達し、AI にその場に即した演技を促します。

---

## 2. 出力形式の強制：JSON フォーマットの「死守」

バックエンドのプログラムにとって、AI の生成内容がいかに美しくても、フォーマットが崩れていればエラーの原因となります。

### 🛡️ 堅牢性を高める 3 つの防衛策

1.  **One-Shot Example (一撃必殺の範例)**: 言葉で説明するだけでなく、望ましい JSON 出力の「完全なサンプル」を提示します。余計な説明文を省き、純粋なデータ構造のみを出力させるためのデファクトスタンダードです。
2.  **思考プロセスの隠蔽 (Hidden CoT)**: AI に思考（推論）を促す場合、`<thought>` タグ内で中間の推論を行わせます。API ゲートウェイ側でこの部分を正規表現で除去することで、フロントエンドには純粋なデータのみを渡します。
3.  **セルフヒーリング (自己修復機能)**: 万が一 AI が閉じ括弧を忘れる等のミスをした場合、エラーメッセージを AI に返し、「4 行目のカンマが抜けている、今すぐ修正して再出力せよ」と自己修復を試みさせるリトライ機構を構築します。

---

## 3. 実務で遭遇する「フォーマット崩壊」の回避策

プログラムと AI を連携させる際、特に注意すべき 2 つの「地雷」があります。

- **Markdown 装飾の汚染**: AI は良かれと思って JSON の前後に \`\`\`json タグを付けがちです。これはプログラムのパーサーには不要なため、プロンプトで厳禁するか、処理時に文字列置換で強制的に除去する必要があります。
- **ダブルクォーテーション競合 (Quote Hell)**: JSON の構文記号である「"」と、セリフとしての「"」が衝突するとパースに失敗します。
  - **改善策**: セリフの部分には **全角のカギ括弧「 」** を使用するよう AI に強制指示します。これにより、技術層とコンテンツ層を視覚的にも構造的にも完全に分離できます。

---

## 💡 実務への応用アドバイス

高度な制御が求められるプロンプトを設計する際は、AI が「それ以外の選択肢を選べない」状態に追い込むことが重要です。

- **制約のネガティブ定義**: 「〜するな」という禁止事項よりも、「〜せよ」という肯定的な制約を具体化し、出力の自由度を適切に制限することで、システムの安定稼働を実現してください。
