# 巨石の崩壊と再編：サーバー状態キャッシュとマイクロフロントエンド（State & Micro-Frontends）

## @Overview

こんにちは、AKIRA です。
今回は、フロントエンドプロジェクトが小さくかわいいものから何十万行ものコードを持つ「巨石モンスター」に膨れ上がったとき、何が起きるかについてお話しします。

こんなコードを見たことはありませんか？サイドバーの開閉を管理する True/False 変数が、1万件の注文データを取得する API 配列と一緒に、無理やり一つのグローバル状態管理ライブラリ（Vuex や古い Redux）に押し込まれているコードを。今日は「なんでも Redux に詰め込む」暗黒時代の迷信を打ち破り、大手企業がどのように「マイクロフロントエンド（Micro-Frontends）」で異なるチームをパズルのように組み合わせているかを見ていきます。

## 1. 一線を引く：サーバー状態 vs クライアント状態

かつてのフロントエンドには、あらゆるデータをグローバル Store に詰め込もうとする強迫観念がありました。これが史詩級の惨事を引き起こします：サイドバーを開閉するだけで、グローバルエンジンが監視メカニズムにより 1万件の注文データも変化していないか走査してしまい、パフォーマンスが連続ヒットで瀕死状態に。しかもその 1万件の注文は「昨日」のデータで、ユーザーは期限切れの画面を見ながら気づいていない。

現代の高度なアーキテクトは、心の中に厳格な楚河漢界（絶対的な一線）を引きます：

### 📱 クライアント状態（Client State）

- **これは何？** ブラウザのローカルにのみ存在する「一時的・短期的」インタラクションデータ。
- **例えば**：今はダークモード？フローティングウィンドウ（Modal）は開いている？ドロップダウンで「東京 → 大阪」を選択した。
- **どこに置くべき？** Vue 自身の `ref/reactive` に置くだけで十分。複数コンポーネントをまたぐ共有が必要な場合のみ、純粋な `Pinia` や `Zustand` のような超軽量マネージャーに。ユーザーが F5 リフレッシュしてこれらの設定が消えても問題ない。

### ☁️ サーバー状態（Server State）

- **これは何？** 覚えておいてください：**このデータの所有権は「リモートデータベース」にある！** フロントエンドはただ「ちらりと見る」だけ。あなたに勝手に変える権限はなく、リモートでいつでも誰かに変更される可能性があります。
- **例えば**：商品の最新在庫数、あなたの口座残高。
- **業界の救世主（TanStack Query / SWR）**: もはや自分で API リクエストを手書きして Pinia に詰め込むことはしません。業界は `vue-query` のようなデータフェッチング専門ツールに全面転向しました。

このツールがバックグラウンドで超賢いキャッシュプールを構築してくれます：

- **自動バックグラウンド更新**: LINE を確認してからブラウザに戻ってきた瞬間（Window Focus）、バックグラウンドで密かに API を叩いて商品在庫を最新状態に更新します。
- **壮絶なプリフェッチ（Prefetch）**: マウスカーソルが「次のページ」ボタンに触れた瞬間、まだクリックもしていないのに！200ミリ秒前倒しで次ページのリストを取得しています。海外の達人のサイトでページ切り替えが魔法のように滑らかなのはこれが理由です。
- **楽観的更新（Optimistic UI）**: 記事にいいねをタップしたとき、サーバーから「OK、保存しました」という返事を待ちません。「もう成功した」ふりをして、いいねを瞬時に赤く変えます。1秒後に通信失敗がわかれば、そっとグレーに戻すだけ。これが Facebook のスムーズな体験の秘密です。

## 2. チームの競合を解決する究極の兵法：マイクロフロントエンド

会社が急成長して、フロントエンドエンジニアが 5名から 50名に拡大したとします。全員が巨大な Vue リポジトリ（Monorepo）の中で戦っています。A チームがトップページのチャットボタンを変更したら、B チームの決済カートがサイト全体でビルドエラーを起こしてクラッシュ。Webpack のコンパイルを待つだけで 10分かかり、これでは開発になりません。

### 🧩 積み木理論とモジュールフェデレーション（Module Federation）

フロントエンドアーキテクチャはバックエンドの「マイクロサービス（Microservices）」の概念を盗用し、**「マイクロフロントエンドアーキテクチャ」** を生み出しました。

これにより：

- A チームが【ホームページ】を担当し、最新の Vue 3 で書く。
- B チームが【決済システム】を担当し、歴史的な事情から古い React 16 を維持。

Webpack 5 の **Module Federation（モジュールフェデレーション）** という黒魔術、または **Single-SPA** フレームワークを使って奇跡が起きます：

1.  **独立してデプロイ**: B チームが今日 React のバグを修正。A チームに関係なく、「決済部分のみ」をコンパイルして CDN にアップロード、5秒でデプロイ完了。
2.  **完璧なシームレス統合**: ユーザーがホームページを開くとき、ブラウザはリアルタイムでネットワークから A チームの Vue モジュールを取得し、次に B チームの React モジュールも取得。同一画面上で、奇跡のように無縫合体されてユーザーに届きます—背後で 2つの異なるフレームワークが動いているとは気づきません！

## 💡 Vibecoding 成功の秘訣：AI への発注

AI や新人同僚が API データを Pinia に詰め込む巨大アーキテクチャを書き始めたら、即座に制止してください：

> 🗣️ `「待て！この顧客注文データは神聖不可侵の【サーバー状態 Server State】だ！生の axios を watch と組み合わせてグローバル Pinia に詰め込むことは絶対に禁止！今すぐ【@tanstack/vue-query】を導入しろ！バックグラウンドの自動無効化と再フェッチ（Stale-while-revalidate）、キャッシュ機構、楽観的更新マジックを全部使え。Pinia はローカルのダークモード切り替えのような小事のためだけに残せ！」`

ローカル状態とサーバーキャッシュを分離してこそ、システムのメンテナンスが時限爆弾の解体から解放されます。
