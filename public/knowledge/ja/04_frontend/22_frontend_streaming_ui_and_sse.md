# AI 時代の新通信プロトコル：ストリーミング打字機と SSE イベントストリーム (Streaming UI)

## @Overview

こんにちは、AKIRA です。
今回は ChatGPT の謎を解き明かします。AI がまるでタイプライターのように一文字ずつゆっくりと長文を吐き出すとき、裏側では一体どんな魔法が起きているのでしょうか？

なぜ従来の HTTP のように「データを取得してから一気に表示する」ではダメなのでしょうか？今回は、大規模言語モデル（LLM）が抱える最大の課題である「待ち時間（Latency）」を解体し、なぜ **SSE（Server-Sent Events）の一方向ブロードキャスト** が双方向 WebSocket を打ち破り、タイピングエフェクト（Streaming UI）の王道となったのかを解説します。

---

## 前書き：AI の残酷な待ち地獄

従来の Web API は「一打一答」方式です。ユーザーリストを要求すると、バックエンドが 0.5秒で整理し、まとめて配列で返す。これを Request-Response サイクルと呼びます。

しかし、GPT-4 のような大規模言語モデル（LLM）が相手になると、悲劇が始まります。LLM は「自己回帰モデル（Autoregressive）」です。つまり、**一単語ずつ（トークンずつ）順番にしか計算できません。** 1000文字の長文を書かせようとすると、最後の一文字を生成するまでに、バックエンドが必死に計算し続けること **丸々 15秒** かかることもあります。

もしフロントエンドが従来の HTTP でそれを待ち続けたら、ユーザーのスマートフォン画面は**白いローディングアイコンがぐるぐる回り続けること丸々 15秒！** この 15秒の沈黙は、99% のユーザーに「サイトが落ちた」と思わせ、即座にウィンドウを閉じさせてしまいます。

---

## 1. 体験を救う神技：TTFT とデータストリーム (Streaming)

この発狂しそうな 15秒を解決するため、アーキテクトたちはある重要な指標を提唱しました：
⏱️ **Time-To-First-Token (TTFT、初回単語到達時間)**

人間の心理というのは、実は非常にだましやすいものです。送信ボタンを押してから **0.5秒以内** に、画面上で最初の一文字がタイプライターのように表示され始めると、私たちの不安感は瞬時にゼロになります。たとえその文章が最終的に 15秒かかっても、私たちの脳は「このサイト、爆速だし、AI がまるで考えながら書いてくれてる！」と錯覚するのです。

### 🌊 データストリーム (HTTP/2 Stream Readable)

これを実現するには、HTTP プロトコルの本質を根本から覆す必要があります。バックエンドはもはや全部計算し終わるまで待ってから送信することはしません。GPU が最初の単語 `"The"` を計算した瞬間、バックエンドは水道管から水が漏れるように、その `"The"` をネットワークパケットから即座にフロントエンドへ「押し出します」。そして一語計算するたびに一語押し出します。フロントエンドはトークンを受け取ると、即座に画面上に連結表示します。これがストリーミング（Streaming）です。

## 2. ツール選定戦：なぜ SSE で WebSocket ではないのか？

前の章では、リアルタイムメッセージングのために HTTP ポーリングを捨てて WebSocket（双方向永続接続）を使うべきだと教えました。では、なぜ WebSocket を LLM のタイピングストリームに使わないのでしょう？

| 通信プロトコル                | 特性と結論                                                                                                                                                        | LLM ストリーミングに向かない理由                                                                                                                                                                                     |
| :---------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **WebSocket**                 | **全二重（双方向）**に対応。サーバーとフロントエンドが 24時間互いに叫び合える。ただし Redis Pub/Sub を導入して複雑な接続状態を管理する必要あり。                  | **大げさすぎる！** よく考えれば、AI に質問した後は、**AI が一方的に文字列を流してくるだけ。フロントエンドが返答する必要はありません。** 双方向接続は重すぎる上、流行の Serverless アーキテクチャとも相性が最悪です。 |
| **SSE**（Server-Sent Events） | **単方向ブロードキャスト（サーバー→クライアント）**に対応。本質は純粋な HTTP（ブラウザネイティブの EventSource API ベース）。受信専用のチャンネルを開くイメージ。 | **完璧にマッチ！** 超軽量で、あらゆる従来型プロキシやファイアウォールとも互換性あり。Vercel AI SDK も、OpenAI 公式の Streaming API も、その底層実装は全て SSE 規格です！                                             |

_（備考：React/Next.js などのモダンフレームワークでは、`fetch` API + `ReadableStream` で同じ精神を下位レイヤーで再現することも一般的です。）_

## 3. フロントエンドのバイト処理地獄：デコードと Markdown レンダリング

トークン（字粒）を受け取ったら、そのまま HTML に表示できると思っていましたか？地獄はここから始まります。

1. **文字化けの悪夢**: バックエンドから届くのは純粋なテキストではなく、バイト化された Stream です。`Uint8Array` を UTF-8 テキストに変換（デコード）する必要があります。稀な文字（CJK 文字等）では、一文字が 2つのチャンクに分割されて届くことさえあります！フロントエンドで手動バッファリングして結合デコードしなければ、画面に文字化け「`□□□`」が出現します。
2. **チラつくテーブル（Markdown レンダリング）**: LLM はテーブルや太字を含む Markdown 形式を生成するのが大好きです。しかしテーブルが途中で止まった状態（`| カラム1 | カラム2`）で、普通の Markdown パーサー（`marked.js` 等）に渡してしまうと、エラーでクラッシュするか、画面が狂ったように何度も再レンダリングを繰り返します。ここでは **インクリメンタルレンダリング（Streaming Parsing）** に対応した特殊なレンダリングエンジンの導入が必須です。
