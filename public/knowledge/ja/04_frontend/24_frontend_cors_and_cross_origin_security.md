# ウェブを守るアズカバン：CORS クロスオリジン セキュリティと不正アクセス防衛（CORS）

## @Overview

こんにちは、AKIRA です。
今回は、すべてのフロントエンドエンジニアがキャリアで最初にぶつかる「地獄級エラー」を解体します：**CORS エラー** とは一体何なのか？

この記事では CORS にまつわる神話（エンジニアいじめの定番）を完全に解体し、この戦争で本当の被害者は誰なのか、そして不正なクロスオリジンリクエストや CSRF（クロスサイトリクエストフォージェリ）攻撃を正しくブロックするためのホワイトリスト設定方法まで解説します。

---

## 前書き：なぜ API は Postman では動くのに Chrome では死ぬのか？

無数の新人フロントエンドエンジニアが、期待に胸を膨らませて `fetch("https://api.github.com/...")` と書き込み、F12 を押して成果を収穫しようとすると、恐ろしい赤文字の画面に出会います：
`Access to fetch at '...' from origin '...' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.`

新人はこの赤文字をスクリーンショットして、怒りをぶつけにバックエンドへ：「あなたの API、壊れてるよ！」
ベテランのバックエンドは冷笑しながら Postman を開き、全く同じ URL を叩きます：「どこが壊れてるの？俺の Postman にはデータが満杯だよ？フロントの Bug じゃないの？」

**真実は：どちらも壊れていない。これはすべて Chrome ブラウザが設けた「保護レベルの結界」なのです。**

---

## 1. 原罪：同一オリジンポリシー（Same-Origin Policy, SOP）

インターネットには極めて古くて融通の利かないルールがあります。「同一オリジンポリシー」です：**「Aドメインのウェブページは、Bドメインの API が返したデータを絶対に盗み見てはならない」**
（しかもその基準は非常に厳格で、プロトコル `http/https`、ドメイン `moyin.com`、ポート `443`、この3つのうち1文字でも違えばブラウザはクロスオリジンと判定します！）

**このルールがなかったらどうなるでしょう？**
例えばあなたがハッカーの悪意あるサイト `hacker.com` を閲覧しているとき、ハッカーがバックグラウンドにこんな JavaScript を仕込んでいたとします：`fetch("https://bank.com/api/my_balance")`。あなたのブラウザはすでに `bank.com` に Cookie でログイン済みです。もし同一オリジンポリシーという壁がなければ、ハッカーのスクリプトは**あなたの銀行残高、さらには個人情報まで、全部抜き取って自分のページに表示できてしまいます！**

Chrome や Edge のようなブラウザは、あなたの銀行データを守るため、ハッカーのスクリプトが `bank.com` の返ってくる残高を覗こうとした瞬間、その情報を「ボディーガードのように死守して絶対にハッカーのスクリプトには渡さず」、F12 に赤い CORS エラーを表示します。
**これだけは死んでも覚えてください：CORS エラーはユーザー（フロントエンドクライアント）を悪意あるフィッシングサイトから守るためのものです。サーバーは守っていません！サーバーはすでにデータを返しています。それを偉大な Chrome が遮断しているだけです！**

---

## 2. 結界を越えるビザ：CORS（クロスオリジンリソース共有）

しかし現代のフロントエンド/バックエンド分離アーキテクチャでは、フロントが `www.moyin.com` にある一方で、バックエンド API は `api.moyin.com` にあります。ブラウザ視点では依然としてクロスオリジンです！自分たちのサイトへのアクセスまでブロックするわけにはいきません。ここでバックエンドエンジニアが「CORS ビザ」を発行する必要があります。

### 🛑 プリフライトリクエスト：探索の OPTIONS パケット（Preflight）

フロントエンドが攻撃的な、またはデータ変更を伴う操作（例：JSON や Authorization Token 認証を含む `POST` リクエスト）を `api.moyin.com` に送ろうとする直前、Chrome ブラウザは密かに `OPTIONS` という幽霊探索リクエストを API サーバーへ先に送ります：
_「こんにちは！私は今 `www.moyin.com` にいますが、後で `POST` でデータを送ってもいいですか？」_

### 🎫 バックエンドの合言葉（Headers）

バックエンドの API サーバー（Nginx や Node.js 等）はこの `OPTIONS` 探索ボールを受け止め、次の合言葉を含む HTTP Headers を返さなければなりません：

- `Access-Control-Allow-Origin: https://www.moyin.com`（このドメインからしか受け付けない！）
- `Access-Control-Allow-Methods: GET, POST, OPTIONS`（これらのメソッドのみ許可）

Chrome はこのビザを受け取り、ドメインが一致することを確認します。「ああ、自分たちのものだ」と判別して、はじめて抑留していた本物の `POST` リクエストを正式に開放します！

## 3. 不正クロスオリジンアクセス防衛のベストプラクティス

初心者バックエンドエンジニアの多くが、開発の利便性を優先してこのような世紀末 CORS 設定を書いてしまいます：

```http
Access-Control-Allow-Origin: *
# (ワイルドカード：つまり全宇宙のウェブサイトがここに来てデータを自由に取得できる！)
```

**会員ログインシステムのある API でこれをやると、自分の金庫の扉を全世界のハッカーに向けて全開にするのと同じです。**

### 🛡️ 滴水不漏の防衛マトリックス

アーキテクトとして、次の点を必ず守ってください：

1.  **厳格なホワイトリスト配列**: バックエンドコードに `ALLOWED_ORIGINS = ['https://moyin.com', 'https://admin.moyin.com']` を必ずハードコードします。リクエスト元がリストにない場合、容赦なく `HTTP 403 Forbidden` で門前払いです。
2.  **認証情報付与の禁止（Credentials）**: 本当に必要でない限り、`Access-Control-Allow-Credentials: true` を安易に有効化してはいけません。これにより外部サイトからの API リクエストに「ユーザーがあなたのサイトでログインした Cookie」が自動付与されます。これは **CSRF（クロスサイトリクエストフォージェリ）** 攻撃の防止に関わる一線です。
3.  **SameSite Cookie による究極ロック**: Cookie で JWT 認証を管理するなら、Cookie 属性を必ず `SameSite=Strict`、または少なくとも `Lax` に設定してください。こうすることで外部サイトから発行された API リクエストに対して、ブラウザは物理的にログイン Cookie を付与しなくなり、クロスオリジン攻撃の可能性を根本から断ちます。

## 💡 Vibecoding 成功の秘訣：AI への発注

> 🗣️ `「AI 助教！よく聞け。この Express バックエンドの CORS チェック層（Middleware）を設定する際、Access-Control-Allow-Origin: * のようなセキュリティ意識ゼロのガレージコードを書くことは厳禁だ！
正規表現またはホワイトリスト配列を使った厳格な CORS 設定ロジックを構築し、我々の【本番ドメイン】と【Vercel プレビュードメイン（Preview Deployments）】からのフロントエンドへのアクセスのみを許可しろ！
OPTIONS プリフライトリクエストに対しても最小権限の Allow-Methods で対応し、フロントエンド開発の利便性のためにドメイン全体のセキュリティを犠牲にする愚を絶対に犯すな！」`

CORS の結界を守って初めて、合格の Web エンジニアと名乗る資格があります。
