# DOM の限界を救済し Google 神を喜ばす：フロントエンドパフォーマンスと仮想リスト（Performance）

## @Overview

こんにちは、AKIRA です。
今回は多くの初心者フレームワーカーの幻想を壊しに来ました。Vue や React の Virtual DOM を使えばパフォーマンスが最強になると思っている人が多いですが、大間違いです。

社長から株価監視システムを作れと言われ、あなたは無邪気に `v-for="item in 100000"` と書いて、10万件の株式明細を一度に画面に表示しようとします。次の瞬間、Chrome が悲鳴を上げて「メモリ不足、タブクラッシュ」を送りつけてきます。今日はこの物理的な限界と、伝説の「目くらまし手品」でいかにそれを克服するか、さらに米国 EC 市場の生殺与奪を握る Google のコアウェブバイタル（Core Web Vitals）についてお話しします。

## 1. 目くらましの極致：仮想化リスト（Virtualization / Windowing）

なぜ 10万件のデータがブラウザをクラッシュさせるのか？Virtual DOM がどれだけ優秀でも、最終的には本物の DOM ツリー上に 10万個の本物の `<div>` ノードを誠実に構築しなければならないからです。ノードが 1,500個を超えると、CSS レンダリングとイベントバインディングの効率がスキー場の急斜面を滑り落ちるように崩壊します。

この課題に対して、大手企業の本当の解法は非常に直接的で強引です：**「騙す！」**

よく考えてみてください。底に 100万件のデータが隠れていても、スマートフォンの画面はあんなに小さい—**ユーザーが今「目で見える」のはせいぜい 20件だけです！**

### 🪟 ウィンドウ切り分け術（vue-virtual-scroller を例に）

この黒魔術の仕組み：

1.  **偽スクロールバー**: フレームワークはあなたに 10万件あることを知っています。1件 50px なら合計 500万 px。隠し場所に空のボックスを置いて 500万 px の高さの偽骨格を作ります。右端のスクロールバーが極細に見える完璧な偽装です。
2.  **20個の苦役だけ**: これが最も恥知らずな部分です。システムは実際のウェブページ上に永遠に 20個の `<div>`（プラスバッファ数個）しかマウントしません。
3.  **無限なる瞬間切り替え**: 猛烈にスクロールして 200件目の商品に到達したとき、フレームワークは新しい要素を作りません！テーブルにある 20個の `<div>` の中のテキストと画像を、**瞬時に荒々しく【201〜220件目】の内容に差し替えます**！そして CSS の `transform: translateY` で瞬間移動させ、あなたの視線が止まったその画面にピタリと貼り合わせる！

わかりますか？10万件でも 1000万件でも、世界にはその 20数個の可哀想な DOM 要素が延々とレンガ運びのような循環作業をするだけ。時間計算量が永遠に $O(1)$ を維持する神の領域です。

## 2. Google の生死を判定する閻魔：コアウェブバイタル（Core Web Vitals）

データ処理の話が終わったら、もう一つの残酷な現実を語りましょう。自分が書いたウェブページが他人のスマートフォンを熱くさせても気にしないかもしれませんが、Google 神だけは絶対に怒らせてはいけない。2021年から Google は 3つの法宝「**コアウェブバイタル**」を SEO の生殺与奪権として握りしめました。この 3指標が赤点不合格なら、どんなに良い記事を書いても検索順位は地の底まで強制降格されます。

### 🥇 LCP（最大コンテンツの描画 — Largest Contentful Paint）

- **何を測る？** ページで最大のコンテンツ（例：メインのビッグポスター、最大の商品タイトル）が画面に現れるまでの時間。
- **制限時間**: **2.5秒** の生死ライン！
- **救済法則**: ヒーローの大きい画像に `background-image` CSS を使うのは厳禁！（CSS の読み込みが終わって初めて画像取得が始まるので遅すぎる！）素直に HTML の `<img fetchpriority="high">` を使い、ブラウザに割り込み最優先でこの画像を取得させる。

### 🏎️ INP（次のペイントとのインタラクション — Interaction to Next Paint）

_（これは役立たずだった FID 指標を排除して登場した新王です）_

- **何を測る？** 記事を読んでいたユーザーが「お気に入り」をタップしたとき、ハートが一瞬遅れてから赤くなると不合格。
- **制限時間**: タップしてから **200ミリ秒（0.2秒）** 以内に画面フィードバックが必要！
- **救済法則**: React/Vue 開発者の天敵。タップした瞬間にバックグラウンドで巨大な `for` ループ（長タスクがメインスレッドをブロック）が走っていると画面が脳梗塞のように固まります。汚い仕事は Web Worker にバックグラウンドで計算させるか、前述の**楽観的更新（Optimistic UI）**で先に画面を動かして脳を騙す。

### 🌋 CLS（累積レイアウトシフト — Cumulative Layout Shift）

- **何を測る？** ニュース記事を読んでいて「次のページ」をクリックしようとしたとき、突然上に巨大な広告バナーが読み込まれて記事全体が 5センチ下に押しやられ、うっかりその広告をクリックしてしまう！ユーザー体験の極度の憤慨。
- **制限時間**: 画面全体のシフトスコア累計が **0.1** を超えてはならない（ほぼ静止しているよう求められる）。
- **救済法則**: 強制ルール！ページ上のすべての `<img>` タグは、画像がネットから引っ張られていない段階でも `width` と `height`（または CSS の `aspect-ratio`）を事前に書き込まなければならない。透明なセメントの穴を先に掘って場所を確保することで、画像が遅れて来ても後ろのテキスト全体を崖から突き落とさずに済む。

ビッグデータシナリオや EC サイトのトップページのような高流量エントリーを AI に作らせる際は：

> 🗣️ `「AI 助教！止まれ！この 5万件のバックオフィス注文 DOM ノードをネイティブの v-for で力任せにレンダリングするのは禁止だ！今すぐ【vue-virtual-scroller】を導入するか、ネイティブの Windowing（仮想ウィンドウ）メカニズムを実装しろ！各行の高さは光速の動的入れ替えを実現するために 60px に固定しろ！また、ホームページのヒーローバナーには Google の CLS 審査を満点でパスするための幅高さプレースホルダーを必ず付けて、レイアウト大地震を防げ！」`

これこそアーキテクトに求められる高さと気魄です。
