# Hybrid AI Strategy: Ollama Local Deployment and Cloud Cookie Session Management

## @Overview

When developing an AI engine, you face a fundamental choice: should you use local inference for cost and privacy, or cloud APIs for peak intelligence? In the Moyind project, we don't choose—we use both. This article covers how to deploy local LLMs using Ollama and how to programmatically extract cloud session cookies for seamless, hybrid AI integration.

---

## 1. Local Model Hub: Ollama Deployment Guide

If you own an NVIDIA RTX GPU, you should leverage local LLMs to eliminate API costs and ensure total privacy. **Ollama** is the premier tool for managing these open-source models with a single command.

### 📦 Choosing a Model based on VRAM

Your GPU's memory (VRAM) is the ultimate ceiling for your AI's intelligence:

- **2-4GB (Entry-level)**: Use `llama3.2:3b`. It’s small but sufficient for API structure testing or basic NPC dialogue.
- **6-8GB (Standard Laptop/3060)**: The **Golden Benchmark**. We recommend `qwen2.5:7b`. It offers exceptional logic for its size.
- **12GB+ (High-end)**: Go for `qwen2.5:14b`. Reasoning depth increases significantly at this scale.
- **24GB (Enterprise/RTX 3090/4090)**: You can comfortably run `32b` or larger preview models with near-GPT-4 levels of reasoning.

## 2. Cloud Session Management: The Firefox Cookie Scheme

Sometimes you need the power of a flagship model (like Gemini 1.5 Pro) without the overhead of enterprise API quotas. Programmatically extracting session cookies allows your engine to leverage authenticated web sessions.

### 🦊 Why Firefox?

Google Chrome encrypts cookies using hardware-bound system keys, making them difficult to extract without triggering security alerts. **Firefox**, however, stores session data in a relatively accessible SQLite database.

1.  Log in to the desired AI service via the browser.
2.  Our extraction script locates the Firefox profile and pulls the `Session Cookie` directly from the database.
3.  The API Gateway uses this token to authenticate requests as if they were coming from a legitimate user session.

---

## 3. High-Performance Responses: Native Context API

When using Ollama locally, avoid using "OpenAI-compatible" middleware if speed is your priority. We connect directly to Ollama’s native `/api/chat` protocol to achieve millisecond response times.

- **KV Cache Reuse**: Instead of resending the entire conversation history every turn, we capture the `context` ID returned by Ollama.
- **Performance Hit**: By resubmitting this ID, the model skips re-processing old tokens. Response latency drops from several seconds to under 500ms.

---

## 💡 Practical Engineering Insights

When building a hybrid model access layer:

- **VRAM-Aware Downscaling**: Write logic that detects your available VRAM. If it drops below 6GB, automatically downgrade the default model to `llama3.2:3b` to prevent OOM errors.
- **Stateful Connection Layer**: Use the native `/api/chat` endpoints for local models to maintain state internally, rather than relying on stateless HTTP headers, to ensure steady performance in real-time interactive applications.
