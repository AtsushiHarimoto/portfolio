# Vibe-Coding 實戰工作流：設計、測試、碼入、驗證的黃金循環 (Vibe Coding Workflow)

## @概覽

在「Vibe-Coding」中的開發，並不只是單純地寫代碼，而是指揮 AI Agent 穩定產出「可運作軟體」的過程。

在 Moyin 專案中，我們採用了一套低成本、具可回溯性、且高度可驗證的自動化工作流：**DESIGN ➝ Tests ➝ Code ➝ Verify**。

---

## 1. 工作流的四大支柱

1.  **架構設計 (DESIGN)**：人類與 Reviewer（AI）協同產出權威性的 `2.DESIGN.md`，作為唯一真理來源（SSOT）。
2.  **測試防護 (Test-First)**：在實作前，由 Codex 建構 E2E 測試骨架。這就是這場開發任務的「發包合同」。
3.  **代碼實裝 (Coding)**：Antigravity 依據設計規格，精準地將源碼落地實踐。
4.  **驗收裁決 (Verify)**：以測試結果作為唯一允收標準，直到全數通過（Green State）為止。

---

## 2. 核心紀律

- **唯一真理來源 (SSOT)**：所有開發更動僅依歸於 `2.DESIGN.md`。
- **測試先行**：視測試腳本為「發包合同」，實際代碼實裝則是「履約交付」。
- **小步快跑**：單次任務流僅聚焦於單一功能點或單一缺陷的修復。

---

## 3. 標準化作業流程 (SOP)

### Step 1：架構策劃 (Review)

產出高度可執行的 `DESIGN.md`。除了定義「要做什麼」，更要清晰界定「絕對不做什麼」。涵蓋介面視覺、數據流與商業邏輯。

### Step 2：測試防禦 (Design)

指派 Codex 研讀設計書，並產製 E2E 測試腳本（如 Playwright）。每一項驗收標準都必須對應一個獨立的測試用例。

### Step 3：代碼實裝 (Coding)

指派 Antigravity 依據設計藍圖與測試骨架進行開發。AI 被嚴格限制在設計書所劃定的範圍內。

### Step 4：驗收裁決 (Verify)

於本地觸發自動化測試網。

- 🔴 **失敗**：收集錯誤日誌並丟回給 Antigravity 進行強制修復。
- 🟢 **通過**：予以簽核合併，並發佈成果。

>
> 1. 先跟我對齊並產出權威性的 【2.DESIGN.md】。
> 2. 依據設計，產出 【E2E 測試骨架】，這是你的發包合同！
> 3. 測試準備好後，開始實裝代碼。
> 4. 最後你自己跑測試，給我看到 100% Green State 的證據。
>    不准有任何模糊地帶，現在去跑通這個循環！」`

這套自動化回饋循環，是讓「Vibe-Coding」從口號昇華為實戰開發武器的關鍵。
