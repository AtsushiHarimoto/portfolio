# 07.5. 混合動力戰略：Ollama 本機部署與雲端 Session 提取技巧

## @概覽

在 AI 引擎開發中，我們面臨著一個核心平衡：是要選擇本機推理帶來的零成本與隱私，還是雲端 API 的最強智力？在 Moyind 的架構中，我們採用「混合動力」模式。本篇將介紹如何透過 Ollama 在本機快速部署優質開源模型，以及如何利用 Firefox 提取雲端 Session Cookie 以實現高效率的 API 劫持與整合。

---

## 1. 本機模型中樞：Ollama 部署實務

若您擁有 NVIDIA RTX 系列顯卡，強烈建議利用 **Ollama** 架設本機推論環境。這能讓您在無需支付 API 費用且完全斷網的情況下，依然保有強大的語言處理能力。

### 📦 模型選型與顯存 (VRAM) 策略

顯存的大小決定了您能運行模型的「智商上限」：

- **2-4GB**: 建議運行 `llama3.2:3b`。雖然規模較小，但用於測試 API 結構或基礎 NPC 對答已足夠。
- **6-8GB (主流規格)**: **黃金配置**。推薦選用 `qwen2.5:7b`。阿里的通義千問模型在中文語意與邏輯理解上是目前 7B 等級的首選。
- **12GB+**: 可上調至 `qwen2.5:14b`，推論深度將有顯著提升。
- **24GB (頂規卡)**: 可流暢運行 `32b` 以上甚至預覽版模型，邏輯與記憶力足以媲美旗艦級雲端模型。

---

## 2. 雲端認證劫持：Firefox Cookie 提取方案

當我們需要使用雲端最強模型（如 Gemini 1.5 Pro）進行複雜劇情生成，且希望利用網頁版額度時，如何獲取認證 Token 是關鍵。

### 🦊 為什麼選擇 Firefox？

Chrome 對於 Cookie 的加密受限於系統硬體密鑰，解鎖過程繁瑣且易觸發安全警報。而 **Firefox** 將數據存儲於相對易於訪問的 SQLite 資料庫中：

1.  在瀏覽器中手動登入目標 AI 服務。
2.  透過提取腳本讀取 Firefox Profile 資料夾，從 SQLite 中直接抓取認證 Token。
3.  將 Token 注入 API Gateway，使程式端能模擬網頁端的操作權限，實現高成功率的自動化接入。

---

## 3. 毫秒級響應：原生上下文快取技術

在使用 Ollama 時，應優先對接原生 `/api/chat` 協定而非相容層。Moyind 透過以下技術追求極致的流暢感：

- **KV Cache 複用**: 透過維護 `context` ID，我們在下一輪對話中無需重傳歷史對話，僅需傳送增量訊息。
- **性能提升**: 這使推論啟動延遲從數秒降低至 500ms 以內，實現視覺小說那種「秒點秒出」的高度沉浸體驗。

---

## 💡 技術架構師的實務建議

要在專案中實現穩定的混合模型策略：

- **VRAM 負載預警**: 實作顯存偵測邏輯。若可用顯存低於預設閾值，系統應自動將本機模型降級，或將流量導回雲端 API。
- **自動化 Session 維護**: 基於 Firefox 提取技術，建議設定定時任務自動刷新 Token，確保雲端接入層的 24/7 穩定性，免去手動維護的困擾。
