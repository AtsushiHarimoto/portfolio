# 07.6. 支配 AI 的權能：提示詞工程學 (Prompt Engineering)

## @概覽

提示詞工程 (Prompt Engineering) 並非單純的「文學創作」，在專業的 AI 軟體開發領域，它是標準的「工程控制」。大型語言模型 (LLM) 本質上是演技極佳但缺乏常識的「即興演員」。若無精確的指令規範，其產出內容會過於發散，導致系統崩潰。本篇將解析 Moyind 專案如何透過「絕對控制力」打造穩定且商用的提示詞架構。

---

## 1. 提示詞雙軌制：System 與 User 的權責分明

在 API 協議中，指令被切分為兩個完全不同的維度，確保環境約束不被內容污染：

### 👑 System Prompt (全域系統指令)

這是「導演的隱形擴音器」。其埋藏於程式後端，普通用戶無法察覺。

- **權威性**: 權重最高，是 AI 必須死守的底線（如：「你現在是林晚，只能使用全形標點」、「強制回傳 JSON 格式」）。
- **任務**: 錨定人格特徵與技術邊界。

### 🗣️ User Prompt (場景與玩家輸入)

代表「舞台上發生的即時對白」。

- **任務**: 傳達動態的互動狀態。例如：玩家的動作，或系統注入的背景參數（如：【環境參數】深夜、大雨、廢棄校舍）。

---

## 2. 輸出格式的「死命令」：JSON 強制約束

對後端程式而言，AI 的內容再優美，若格式出錯（報錯或多說廢話）則毫無價值。若 AI 在 JSON 前多說一句：「好的，這是預期的內容...」，解析器會直接報錯。

### 🛡️ 實戰防禦三部曲：

1.  **一擊必殺範本 (One-Shot Example)**: 在 System Prompt 中直接提供一個「完美的輸出 JSON 格式」作為範例。這能大幅降低模型出錯的機率。
2.  **思維鏈隱藏術 (Hidden CoT)**: 若 AI 需要推論空間（如判斷是否觸發機關），可要求其在 `<thought>` 標籤內思考。隨後我們在 API 閘道端以正則表達式濾除這段雜質，確保前端僅接收到純粹的數據。
3.  **自癒重試機制 (Self-Healing)**: 當解析發生異常（如括號未閉合），GateWay 會捕獲錯誤並反饋給 AI：「輸出格式錯誤，第 4 行缺少逗號，請修正後立即重傳」。

---

## 3. 常見的格式崩潰雷區 💣

在代碼對接時，必須嚴防以下兩大雷區：

- **Markdown 語法污染**: AI 標籤慣於在 JSON 前後加入 \`\`\`json。這對於後端解析是多餘的，應在 Prompt 中嚴禁，或在代碼端進行物理去除。
- **雙引號地獄 (Quote Hell)**: JSON 語法使用英文雙引號。若劇本對話也使用英文引號，解析將會破裂。
  - **解決方案**: 強制命令 AI 在編寫對話內容時，**一律改用全形中文引號「 」**，實現技術層面與內容層面的完美隔離。

---

## 💡 工程監工的經驗談

好的提示詞工程，應讓 AI 覺得它別無選擇，只能精準執行。

- **肯定性約束**: 與其告訴 AI「不要做什麼」，不如提供具體的「模板」強迫它照著做。降低輸出的隨機性，是實現 AI 規模化商用的第一步。
