# 07.2. 高階 AI 助手技能擴充規範：目錄結構與軍規級配置

## @概覽

要為 AI 助手（如 Claude Code 或 Antigravity）掛載具備自動化執行能力的「自定義技能」，單純提供一份指令清單是遠遠不夠的。在 Moyind 專案中，我們遵循一套嚴謹的目錄結構與元數據 (Metadata) 標準，確保 AI Agent 的環境感應引擎能精準觸發特定技能。本篇將介紹如何設計具備「封裝性」的企業級 AI 技能包。

---

## 1. 致命的初學者誤區：單檔集中式陷阱

許多開發者傾向於將所有指令寫在單一個大型 Markdown 檔案中。然而，在現代 AI Agent 的掃描機制下，這是一個嚴重的設計失誤。

- **鐵律**: 為了確保 Agent 能精準執行 `@技能名` 的定向觸發，**每一個獨立的技能 (Skill) 都必須擁有其專屬的實體目錄。**
- **優勢**: 將技能目錄化處理，能讓 Agent 在呼叫 `@指令` 的瞬間，一次性打包讀取該目錄內的所有背景資料（如 JSON Schema 或 輸出模型），實現真正的組件化。

---

## 2. 標準化技能目錄結構 (Skills Layout)

為了讓 AI 能在掃描瞬間識別技能，請保持層級結構的一致性：

```text
.agent/skills/          # 專案技能總部
├── handoff/            # 每個技能都是獨立的目錄
│   ├── SKILL.md        # ⬅️ 核心定義文件
│   └── TEMPLATE.md     # 輔助用的模板或參考數據
│
├── new_issue/          # 對應輸入「@new_issue」
│   └── SKILL.md
│
└── dev/                # 對應輸入「@dev」
    └── SKILL.md
```

### 為什麼採用目錄封裝？

高階技能（如 `handoff` 任務交接）通常不只有一段文字指令，還包含特定的輸出格式與參考規範。只有透過目錄封裝，AI 才能在載入技能時，獲取完整的「執行上下文」，這才是開發穩定 AI 系統的底氣。

---

## 3. `SKILL.md`：技能的「數位身分證」

在每個技能目錄中，必須包含一個命名的 `SKILL.md`。該文件最關鍵的部分在於頂部的 **YAML Frontmatter** 區塊，這是 AI 辨識指令的唯一憑據。

```markdown
---
name: new_issue
description: "協助建立新任務追蹤實體。觸發方式：@new_issue"
---
```

### 關鍵成功要素

1.  **`name`**: 必須與資料夾名稱完全一致。
2.  **`description`**: 需簡明扼要。強烈建議在結尾加上「觸發方式：@xxx」，這對 AI 的搜尋引擎來說是非常高權重的語意標籤。

---

## 💡 技術架構師的實務建議

深度的技能包不只是「教學」，而是一套「SOP (標準作業程序)」。在設計時請注意：

- **確定性指令**: 在 `SKILL.md` 中應避免模糊的詞彙，轉而使用精確的操作步驟，引導 AI 產生預期內的行為。
- **同步機制**: 建議在專案中維護一套同步腳本，將 `.agent/skills/` 的變動即時掛載到執行路徑，確保開發環境的「唯一真理來源」。
